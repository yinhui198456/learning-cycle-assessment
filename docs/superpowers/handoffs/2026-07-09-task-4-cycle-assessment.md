## Data model

Add the following directly to `apps/learning/models.py`.

### `LearningCycle`

- `name`: unique, generated by the creation service
- `cycle_type`: `calendar_year` or `rolling_12_month`
- `start_date`, `end_date`
- `status`: `active` or `archived`, default active
- `created_by`: Leader user, `PROTECT`
- `created_at`
- constraints:
  - end date is not before start date
  - calendar-year dates and rolling date calculations are validated in the
    creation form/service

No archive endpoint is needed in Task 4; tests may set archived state directly.

### `CycleParticipant`

- cycle FK, `PROTECT`
- member FK, `PROTECT`
- `created_at`
- unique cycle/member pair

Only active users with the `member` Group can be selected. Reject a new active
cycle for a member when its date range overlaps another active participating
cycle. This keeps “current cycle” singular.

### `Assessment`

- member, cycle, capability item FKs, all `PROTECT`
- nullable `current_level` and `target_level`, each restricted to 0–5
- nullable `gap`, computed as `max(target_level - current_level, 0)` when both
  levels exist; otherwise null
- `priority`: blank, `high`, `medium`, or `low`
- `included`: boolean, default false
- `planned_quarter`: blank, `Q1`, `Q2`, `Q3`, or `Q4`
- nullable `planned_month`: date normalized to the first day of a month
- `version`: positive integer, default 0
- `updated_at`
- unique member/cycle/capability-item tuple
- database range constraints for levels and nonnegative gap

Application services must also require the member to be a cycle participant.

## Cycle creation

Leader-only page: `GET/POST /learning/cycles/`.

Use one Django form:

- cycle type
- calendar year for a calendar-year cycle
- start date for a rolling cycle
- one or more participating members

Rules:

- calendar year produces January 1 through December 31 and a name such as
  `2026 年度学习周期`
- rolling cycle produces an inclusive 12-calendar-month range and a name from
  its dates; use standard-library date/calendar operations
- require the relevant year/start field and ignore neither conflicting field
- reject inactive/non-member participants and overlapping active cycles
- create cycle, participants, and initial assessments atomically
- create one assessment for every active `CapabilityItem`
- Leader-only access is enforced server-side; authenticated non-Leaders get
  403 and anonymous users redirect to login

The page lists existing cycles, dates, type, status, and participant count.

## Assessment services and endpoints

Member page: `GET /learning/assessment/`, optionally
`?cycle=<cycle-id>`. It shows only cycles in which the current member
participates. Default to the active cycle containing today, then the latest
active cycle. If there is no cycle, render an explicit empty state.

On page load, atomically create missing assessments for active capability
items. This supports capability items added after the participant joined.

Group rows by category, level-1 domain, and level-2 domain. Fetch related
objects efficiently; do not issue a query per row.

Single-row save:

`POST /learning/assessment/<assessment-id>/save/`

- accept current/target level, priority, included, planned quarter/month, and
  client `version`
- query by assessment id **and current member** so another member's id returns
  404
- non-member users return 403
- archived cycles return JSON status 409
- version mismatch returns JSON status 409 with error `conflict`, the current
  server version, and current saved values
- invalid input returns JSON status 400 with field errors
- valid input computes gap on the server, increments version using an atomic
  compare-and-update, and returns saved values plus counts
- never trust a posted member, cycle, gap, or capability id

Batch save:

`POST /learning/assessment/batch/`

- accept selected assessment ids and only these optional batch fields:
  `included`, `priority`, `planned_quarter`, `planned_month`
- validate all objects belong to the current member and one active cycle before
  writing; unauthorized ids return 404 and archived cycles return 409
- apply all-or-nothing in a transaction and return updated count

## Dense assessment page

This is the confirmed direction: role workbench styling with a dense table,
not a card per capability.

- show cycle name/date range, assessed count, total count, and included count
- filters: category/domain, suggested level, gap, priority, included, and
  filled/unfilled state
- native `<details>` hierarchy folding for category / first-level /
  second-level sections
- sticky capability-name column
- editable current level, target level, priority, included, quarter, and month
- gap is read-only and updated from the server response
- per-row state: unsaved, saving, saved, failed, conflict
- failed saves retain the entered values and expose a retry button
- conflict state tells the member to refresh; do not silently overwrite
- checkboxes plus one batch toolbar for included/priority/quarter/month
- progress counters update after successful saves
- Arrow Up/Down moves to the same editable column in the previous/next visible
  capability row; native Tab order remains intact
- all controls have labels or accessible names, visible keyboard focus, and
  status text; color is never the only status signal
- JavaScript is progressive enhancement only. Permissions, validation,
  calculations, and writes remain server-side.

Keep styling restrained and desktop-focused. Reuse semantic HTML; no icon
library or asset download.
